# 사용 가이드 (Usage Guide)
이 가이드는 PUBG API를 활용하여 개발할 때 고려해야 할 몇 가지 모범 사례와 제안 사항을 자세히 설명합니다.

## 앱 구조 (App Structure)
API를 사용하는 애플리케이션은 일반적으로 다음과 같은 구성 요소를 갖습니다.

### 클라이언트 측 앱 (A client-side app)
애플리케이션에서 사용자와 마주하는 부분입니다. 사용자 입력을 받아 백엔드에 요청을 보내고, 그 결과를 사용자에게 표시합니다. 이는 독립 실행형 웹사이트, Discord 봇, 모바일 앱 또는 여러분이 생각할 수 있는 그 밖의 어떤 형태로든 구현될 수 있습니다.

### 백엔드 서버 (A backend server)
애플리케이션의 핵심 로직입니다. 클라이언트 측 앱으로부터 요청을 받아 API에 적절한 요청을 보내고, 수신된 데이터를 처리하여 데이터베이스에 저장한 뒤, 이를 다시 클라이언트 측 앱으로 반환합니다.

### 데이터베이스 구조 (A database structure)
수집된 데이터가 저장되는 곳입니다. 애플리케이션의 역할에 따라 플레이어 기록, 매치 전적, 순위표(리더보드) 순위 등 보관이 필요한 모든 데이터가 될 수 있습니다.

### 캐싱 계층 (A caching layer)
이 구성 요소는 백엔드 서버와 API 사이에 위치합니다. API 응답을 메모리에 저장하여 사용자 요청 처리 속도를 높이고 API 호출 횟수를 줄여줍니다. 캐싱이 작동하는 방식에 대한 자세한 내용은 '캐싱 개요' 및 '캐싱 구현' 섹션에서 다루겠습니다.

### API (The API)
스택에서의 시각화를 위해 추가되었으며, 이는 PUBG API를 의미합니다.

## 데이터 사용 의도 (Intended data use)
PUBG API에서 제공하는 데이터는 다양한 방식으로 활용되어 여러 종류의 애플리케이션을 만드는 데 사용될 수 있습니다. 이 데이터의 의도된 사용 목적과, 데이터 및 엔드포인트 구조가 이러한 비전을 어떻게 반영하고 있는지 이해하는 것이 중요합니다.

기본 API 엔드포인트(`players` 및 `matches`)는 모든 데이터에 대한 액세스를 제공하는 것이 아니라 특정 검색을 선호하도록 구조화되어 있습니다. 이것이 API 흐름이 특정 플레이어를 검색하는 것부터 시작하는 이유입니다. 일반적으로 애플리케이션은 모든 플레이어를 검색하려 하기보다는 옵트인(opt-in) 흐름을 따르도록 노력해야 합니다. 이러한 흐름이 예상되는 사용 사례에 어떻게 부합할 수 있는지 보여주는 몇 가지 예는 다음과 같습니다.

* **전적 사이트 및 봇:** 사용자가 자신의 통계와 전적을 보기 위해 플레이어 이름을 입력해야 합니다. 즉, 애플리케이션은 해당 상황이 발생할 때만 API를 쿼리하면 됩니다.
* **순위표 및 코칭 서비스:** 사용자가 사이트에 등록해야 합니다. 즉, 애플리케이션은 알려진 플레이어에 대해서만 쿼리하면 됩니다.

그렇긴 하지만, 히트맵을 제공하는 애플리케이션과 같은 일부 애플리케이션은 여전히 대량의 무작위 데이터를 분석해야 할 수도 있습니다. 이를 위해 당사는 `samples` 엔드포인트에 요청을 보내 획득할 수 있는 무작위 매치 샘플을 매일 제공합니다. 샘플링 비율은 샤드마다 무작위적이고 독립적이므로, 이 데이터는 전체 매치 수나 고유 PUBG 플레이어 수를 추정하는 데 사용할 수 없고 사용해서도 안 되며, 플랫폼별 매치 수나 고유 PUBG 플레이어 수를 비교하는 방법으로 사용해서도 안 됩니다.

## 캐싱 개요 (Caching Overview)
일반적으로 애플리케이션은 가능한 한 동일한 데이터를 두 번 쿼리하는 것을 피해야 합니다. 이때 캐싱이 필요합니다. 백엔드가 API에 요청을 보낼 때, 정해진 기간 동안 결과를 재사용할 수 있도록 결과를 캐시(저장)해야 합니다.

예를 들어, 애플리케이션이 플레이어의 게임 내 닉네임(IGN)을 입력으로 받아 통계 및 매치 기록을 표시한다고 가정해 보겠습니다. 서비스에서 플레이어를 처음 검색할 때 애플리케이션은 해당 데이터를 가져오기 위해 API를 쿼리해야 합니다. 그런 다음 캐싱 계층은 정해진 시간(예: 30분) 동안 해당 API 응답을 보관합니다. 해당 시간 내에 동일한 플레이어를 다시 검색하는 경우, API를 호출할 필요 없이 메모리에서 결과가 반환됩니다. 해당 시간이 지나면 캐시된 데이터가 제거됩니다. 다음에 해당 플레이어를 검색할 때 이 주기(API 쿼리 -> 캐시에 저장 -> 30분 후 제거)가 반복됩니다.

애플리케이션 작동 방식에 따라 백엔드 서버의 결과를 캐시할 기회가 있을 수도 있습니다. 이를 통해 데이터베이스나 기타 스토리지 구조에 대한 요청을 줄일 수 있습니다.

## 캐싱 구현 (Caching Implementation)
이 섹션에서는 API 응답을 상위 수준에서 캐시하는 방법에 대해 다룹니다. 아래에 표시된 코드 예제는 GO 언어로 작성되었으며 오류 처리는 포함되어 있지 않습니다.

먼저 플레이어 객체를 가져오는 예제 함수를 살펴보겠습니다.

```go
func getPlayer(playerName string) playerObject {
    apiResponse := makePlayerNameRequest(playerName) // 주어진 플레이어 이름으로 API 쿼리
    if apiResponse != nil {
        // API에서 플레이어를 찾음
        player := processPlayer(apiResponse) // API 응답을 플레이어 객체로 처리
        return player
    }
}
```

이 함수는 특정 플레이어에 대해 호출될 때마다 API에 요청을 보냅니다. API의 플레이어 기록이 자주 업데이트되지 않는다는 점을 고려하면 이는 매우 비효율적일 수 있습니다. 이를 개선하기 위해 기존 기능에 캐시에서 플레이어를 확인하는 과정을 추가할 수 있습니다. 다음과 같이 수정될 것입니다.

```go
func getPlayer(playerName string) playerObject {
    player := getPlayerFromCache(playerName) // 캐시에서 플레이어 객체 가져오기 시도
    if player != nil {
        return player
    }
    // 캐시에서 플레이어 객체를 찾지 못함
    apiResponse:= makePlayerNameRequest(playerName) // 주어진 플레이어 이름으로 API 쿼리
    if apiResponse != nil {
        // API에서 플레이어를 찾음
        player := processPlayer(apiResponse) // API 응답을 플레이어 객체로 처리
        
        // 다음을 위해 플레이어를 캐시에 저장 (30은 캐시에 남아 있어야 하는 시간)
        storePlayerInCache(player, 30)
        
        return player
    }
    return nil
}
```

## 캐싱 소프트웨어 (Caching Software)
캐싱 소프트웨어의 몇 가지 예는 다음과 같습니다.
* **Memcached** - 다양한 언어의 래퍼(wrapper)를 제공하며 비교적 사용하기 쉬운 캐싱 프로그램입니다.
* **Redis** - 더 많은 유연성과 기능을 제공하는 다소 복잡한 캐싱 소프트웨어입니다.
